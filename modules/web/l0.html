Web Application Attacks

Most breaches start in the browser. Web applications are sprawling, fast-changing, and exposed to the whole internet — which is exactly why they're where so many attacks begin.

Why the web?

Web apps accept untrusted input from anyone, anywhere, and turn it into actions on a server. Every input is a question: "what happens if I send something the developer didn't expect?"

Mindset: Take it one phase at a time — each builds on the last.

The OWASP Top 10

The OWASP Top 10 is the industry's shorthand for the most common web weaknesses — injection, broken access control, misconfiguration, and more. At L0 you only need to recognise the categories and the idea behind each.

Input is the attack surface

Forms, URLs, headers, cookies, APIs — anywhere the app reads something you control is a place to test. The attacker's instinct: trust nothing the app claims, verify everything you can reach.

# Observe how an app responds to unexpected input (lab only)
curl -i "https://lab.example/search?q=test"
curl -i "https://lab.example/search?q=test'"   # does it break?
# A different response can hint at how input is handled
Stay legal: Practise only on intentionally-vulnerable labs (e.g. ones you host yourself). Testing real sites without permission is illegal.

Wrap-up

You now know why web apps are such a rich target, what the OWASP Top 10 represents, and why every input is a potential door. That's the surface of web hacking.
